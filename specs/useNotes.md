## Hook: useNotes

### 1. Responsibility and Purpose
`useNotes` is the single source of truth for all note-related data operations. It exposes a reactive notes list (via `useLiveQuery`) that automatically re-renders consumers whenever the underlying IndexedDB data changes, and a set of async CRUD functions. UI components remain fully agnostic of Dexie.js internals.

### 2. Parameters
| Parameter | Type | Default | Description |
|---|---|---|---|
| `filters.notebookId` | `string` | `undefined` | When provided, only notes belonging to this notebook are returned. |
| `filters.isFavorite` | `boolean` | `undefined` | When provided, only notes matching this favorite status are returned. |

### 3. Return Value
| Field | Type | Description |
|---|---|---|
| `notes` | `Array<Object>\|undefined` | Reactive list sorted by `createdAt` descending. `undefined` while the initial query is in-flight. |
| `createNote` | `Function` | Persists a new note and returns its UUID. |
| `updateNote` | `Function` | Updates mutable fields; never overwrites `id` or `createdAt`. |
| `deleteNote` | `Function` | Permanently removes a note by UUID. |
| `getNoteById` | `Function` | Non-reactive point read by UUID. |
| `error` | `Error\|null` | Last caught error; `null` when no error has occurred. |

### 4. Data Model Invariants
- `id` is a UUID generated by `crypto.randomUUID()` at creation time — immutable.
- `createdAt` is set once at creation time — immutable, never overwritten by `updateNote`.
- `updatedAt` is set at creation and refreshed on every `updateNote` call.
- Default values at creation: `notebookId: null`, `isFavorite: false`, `tags: []`.

### 5. Behavioral Specifications (BDD)

```gherkin
Feature: useNotes Hook CRUD and Reactivity

  # ── Initial state ──────────────────────────────────────────────────────────

  Scenario: Hook initializes with an empty notes list
    Given the hook is mounted with an empty database
    When the initial query resolves
    Then notes should be an empty array

  # ── createNote ──────────────────────────────────────────────────────────────

  Scenario: createNote persists the note and makes it available reactively
    Given the hook is mounted
    When createNote is called with a title
    Then the notes array should contain exactly one note
    And the note's title should match the provided value

  Scenario: createNote returns the UUID of the new note
    Given the hook is mounted
    When createNote is called
    Then the returned value should be a non-empty string (UUID)

  Scenario: createNote applies default field values
    Given the hook is mounted
    When createNote is called without explicit notebookId, isFavorite, or tags
    Then the persisted note should have notebookId=null, isFavorite=false, tags=[]

  Scenario: createNote sets createdAt and updatedAt to the current timestamp
    Given the hook is mounted and the system time is fixed
    When createNote is called
    Then the persisted note should have createdAt equal to the mocked timestamp
    And updatedAt should equal createdAt on a fresh note

  Scenario: Notes are returned sorted by createdAt descending (most recent first)
    Given the hook is mounted
    When two notes are created at different timestamps
    Then notes[0] should be the most recently created note

  # ── updateNote ──────────────────────────────────────────────────────────────

  Scenario: updateNote persists changed fields to IndexedDB
    Given a note exists in the database
    When updateNote is called with a new title
    Then the note's title in the database should reflect the update

  Scenario: updateNote stamps a new updatedAt timestamp
    Given a note exists with a known updatedAt value
    When updateNote is called at a later point in time
    Then the note's updatedAt should be greater than the original value

  Scenario: updateNote never overwrites the id or createdAt fields
    Given a note exists with a known id and createdAt
    When updateNote is called with a payload that includes id and createdAt
    Then the note's id and createdAt in the database should remain unchanged

  # ── deleteNote ──────────────────────────────────────────────────────────────

  Scenario: deleteNote removes the note from the reactive list
    Given a note exists in the database
    When deleteNote is called with the note's id
    Then the notes array should be empty

  Scenario: deleteNote permanently removes the note from IndexedDB
    Given a note exists in the database
    When deleteNote is called with the note's id
    Then getNoteById should return undefined for that id

  # ── getNoteById ─────────────────────────────────────────────────────────────

  Scenario: getNoteById returns the correct note by id
    Given a note has been created
    When getNoteById is called with the note's id
    Then the returned object should have the same title as the created note

  Scenario: getNoteById returns undefined for a non-existent id
    Given the database is empty
    When getNoteById is called with any id
    Then the result should be undefined

  # ── Filtering ───────────────────────────────────────────────────────────────

  Scenario: Filtering by notebookId returns only matching notes
    Given two notes exist with different notebookId values
    When the hook is mounted with filters.notebookId set to one of them
    Then only the note belonging to that notebook should be in the list

  Scenario: Filtering by isFavorite returns only matching notes
    Given two notes exist where one is a favorite and one is not
    When the hook is mounted with filters.isFavorite=true
    Then only the favorited note should be in the list

  # ── Error handling ──────────────────────────────────────────────────────────

  Scenario: createNote sets the error state when the DB write fails
    Given the DB write operation is configured to throw an error
    When createNote is called
    Then result.error should be an instance of Error
    And the error message should match the thrown error
```
